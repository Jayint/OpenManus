## 简介
OpenManus 是一个开源的 AI 代理框架，旨在为开发者提供一个灵活、可扩展的平台来构建和定制智能代理。我们欢迎所有社区成员参与项目的发展，无论是通过代码贡献、文档改进还是社区讨论。本指南旨在为贡献者提供清晰的指导，确保贡献过程高效、友好且符合社区标准。

## 代码贡献流程

### 分支管理
我们采用标准的 Git 分支管理策略。主分支（main）用于发布稳定版本，所有新功能和修复应在独立的特性分支（feature branch）上开发。分支命名应简洁明了，反映其功能或修复内容，例如 `feature/data-analysis` 或 `fix/login-bug`。

### 代码风格
代码应遵循 PEP 8 风格指南。我们使用 `pre-commit` 工具来自动化代码格式化和检查。在提交代码前，请确保已安装并运行 `pre-commit` 钩子。

### 测试要求
所有新功能和修复都必须附带相应的测试用例。测试应覆盖主要功能路径和边界情况。我们使用 `pytest` 框架进行测试，测试文件应放置在 `tests` 目录下相应的子目录中。

### 提交规范
提交信息应清晰、简洁，遵循约定式提交（Conventional Commits）规范。提交信息应以动词开头，描述更改的内容，例如 `fix: 修复登录页面的样式问题` 或 `feat: 添加数据可视化功能`。

## 问题报告最佳实践

### 提供可复现的错误报告
当报告问题时，请提供足够的信息以便维护者能够复现和诊断问题。这包括：
- 详细的错误描述
- 复现步骤
- 相关的日志输出或截图
- 使用的环境信息（操作系统、Python 版本、依赖库版本）

### 使用正确的渠道
请在 GitHub 仓库的 Issues 页面提交问题报告。确保选择正确的标签（如 bug、feature request 等）以便问题能够被正确分类和处理。

## 文档改进途径

### 文档格式要求
文档应使用 Markdown 格式编写，确保语法正确、格式统一。文档中应包含清晰的标题、段落和列表，必要时可使用代码块和图表来增强可读性。

### 文档内容更新
我们鼓励社区成员更新和改进文档。无论是修正拼写错误、补充缺失信息还是重写模糊的段落，所有贡献都欢迎。文档文件位于项目的根目录和各个模块的子目录中，如 `README.md` 和 `app/tool/chart_visualization/README.md`。

## 社区参与方式

### 讨论区
我们使用 Discord 作为主要的社区讨论平台。您可以在 Discord 的 #general 频道中提问、分享经验和参与讨论。请遵守社区准则，保持尊重和专业。

### 会议
定期的社区会议将通过视频会议工具举行，讨论项目进展、规划未来方向和解决关键问题。会议时间和议程将提前在 Discord 和 GitHub 上公布。

### 协作开发
我们鼓励社区成员参与协作开发。您可以通过 GitHub 的 Pull Requests 提交代码，参与代码审查，或在 Issues 中讨论技术方案。维护者将定期审查和合并贡献。

## 行为准则

### 我们的承诺
作为成员、贡献者和领导者，我们承诺让社区成为一个无骚扰的环境，无论年龄、体型、可见或不可见的残疾、民族、性别特征、性别认同和表达、经验水平、教育、社会经济地位、国籍、外貌、种族、种姓、肤色、宗教或性取向。

### 我们的标准
促进积极环境的行为包括：
- 对他人表现出同理心和善意。
- 尊重不同的意见、观点和经历。
- 给予并优雅地接受建设性反馈。
- 对自己的错误承担责任并向受影响的人道歉，从中学习。
- 关注对整体社区最有利的事情，而不仅仅是个人利益。

不可接受的行为包括：
- 使用性化语言或图像，以及任何形式的性关注或性暗示。
- 嘲讽、侮辱性或贬损性评论，以及人身或政治攻击。
- 公开或私下骚扰。
- 未经明确许可发布他人的私人信息，如物理地址或电子邮件地址。
- 其他在专业环境中可能被视为不适当的行为。

### 执行责任
社区领导者负责澄清和执行可接受行为的标准，并对任何他们认为不适当、威胁、冒犯或有害的行为采取适当和公平的纠正措施。

### 报告机制
滥用、骚扰或其他不可接受行为的实例可以报告给负责执行的社区领导者，邮箱为 mannaandpoem@gmail.com。所有投诉将被及时和公平地审查和调查。
